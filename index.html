Git Flow

A structured branching model for teams — covering the core concepts,
branch types, commands, and best practices.

main develop feature/* release/* hotfix/*

What is Git Flow?

Git Flow is a branching model introduced by Vincent Driessen in 2010.
It defines a strict branching structure designed around project releases, giving development teams a robust framework for managing features, releases, and bug fixes.

Core idea

  • 🌿Two long-lived branches: main & develop
  • 🔀Short-lived support branches for work-in-progress
  • 🔒main only ever holds production-ready code
  • 🚀Releases are isolated, tested, and tagged

When to use it

  • 📦Versioned / scheduled releases
  • 👥Teams with multiple parallel features
  • 🛡️Projects needing strict production stability
  • 🏗️Long-running support of older versions

The Branch Map

main — production develop — integration feature/* release/* hotfix/*

Core Branches

main

Production branch

  • 🔒Always reflects a production-ready state
  • 🏷️Every commit on main is tagged with a version number
  • 🚫Direct commits are never made here
  • ⬅️Only receives merges from release or hotfix
develop

Integration branch

  • 🔀Central hub where features are integrated
  • 🧪Contains the latest delivered development changes
  • 🌱All feature branches are cut from develop
  • ✅When stable, it is merged into a release branch
# initialise a repo with Git Flow
git flow init
# or manually:
git checkout -b develop main

Feature Branches

Used to develop new features for an upcoming release. Feature branches exist only as long as the feature is in development.

Rules

  • 🌿Branch off from: develop
  • 🔁Merge back into: develop
  • 📛Naming: feature/<name>
  • 🤝Use pull requests for code review before merge
  • 🗑️Delete after merging
# — git-flow CLI —
git flow feature start user-auth
# ... do your work ...
git flow feature finish user-auth

# — manual —
git checkout -b feature/user-auth develop
# ... commits ...
git checkout develop
git merge --no-ff feature/user-auth
git branch -d feature/user-auth

Release Branches

Supports preparation of a new production release. Only release-oriented tasks go here — bug fixes, documentation, version bumping. No new features.

Rules

  • 🌿Branch off from: develop
  • 🔁Merge into: main AND develop
  • 📛Naming: release/<version>
  • 🏷️Tag the commit on main with the version
  • ✅Allows final QA and bug-fixing in isolation
# — git-flow CLI —
git flow release start 1.0.0
# bump version, fix release bugs ...
git flow release finish 1.0.0

# — manual —
git checkout -b release/1.0.0 develop
# ... version bump, minor fixes ...
git checkout main
git merge --no-ff release/1.0.0
git tag -a v1.0.0
git checkout develop
git merge --no-ff release/1.0.0

Hotfix Branches

Used to quickly patch production bugs. The only branch that cuts directly from main, so a fix can be deployed without waiting for the next release cycle.

Rules

  • 🌿Branch off from: main
  • 🔁Merge into: main AND develop
  • 📛Naming: hotfix/<version>
  • 🏷️Bump the patch version & tag on main
  • ⚡Unblocks the production team immediately
# — git-flow CLI —
git flow hotfix start 1.0.1
# ... fix the bug ...
git flow hotfix finish 1.0.1

# — manual —
git checkout -b hotfix/1.0.1 main
# ... fix ...
git checkout main
git merge --no-ff hotfix/1.0.1
git tag -a v1.0.1
git checkout develop
git merge --no-ff hotfix/1.0.1

Lifecycle Cheatsheet

SETUP 
git flow init
# follow the prompts
FEATURE 
git flow feature start name
git flow feature finish name
git flow feature publish name
git flow feature pull origin name
RELEASE 
git flow release start 1.0
git flow release finish 1.0
git push origin --tags
HOTFIX 
git flow hotfix start 1.0.1
git flow hotfix finish 1.0.1
git push origin --tags

Benefits & Trade-offs

✅

Clean history

Merge commits with --no-ff preserve branch context in the log.

🛡️

Production stability

main is always releasable; hotfixes can ship without touching in-progress work.

👥

Parallel development

Multiple feature branches can coexist without blocking each other.

📋

Clear release pipeline

Dedicated release branches give QA a stable target for final testing.

⚠️

Can feel heavy

Overkill for small teams or continuous-delivery products — consider GitHub Flow instead.

🔁

Long-lived branches = merge conflicts

The longer a feature branch lives, the more divergence accumulates — merge often.

Summary

  • 🟢main  →  always production-ready, tagged at each release
  • 🔵develop  →  integration point for finished features
  • 🟣feature/*  →  one branch per feature, off develop
  • 🟡release/*  →  freeze features, finalize & tag, merge both ways
  • 🔴hotfix/*  →  emergency fix off main, merge both ways

Git Flow shines for versioned, scheduled releases.
Pair it with pull requests and CI/CD for best results.
